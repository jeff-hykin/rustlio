# Relocalization benchmark — BASELINE (current C++ ICPLocalizer)

Snapshot of the **current** relocalizer (`localizer/src/localizers/icp_localizer.cpp`,
two-stage point-to-point PCL ICP), captured *before* any improvements. This is
the number to beat — for both the planned tweaks and the future Rust port.

- Algorithm: two-stage point-to-point ICP (rough 0.25m/5it, refine 0.1m/10it), default config
- Generated by: `./run/reloc_test --json reloc_bench/results/baseline.json`
- Machine: Apple Silicon (arm64), PCL via Homebrew
- Metric: per initial-guess-error bucket — `correct%` = recovered pose within 0.30 m & 5° of truth
- Machine-readable: `baseline.json`

> Regenerate any time with `./run/reloc_test`; compare against this file. Scenarios
> are seeded/deterministic, so re-runs are stable (timings vary with the machine).

## hk_village3 (real Go2 LiDAR)

```
guess err    n  conv%  correct%  te_med  te_p90  re_med  re_p90  ms_med
-----------------------------------------------------------------------
exact       39   100%       87%   0.056   0.482    0.12    1.23   399.6
near       117    95%        0%   2.056   5.745   10.47   20.09   405.8
mid        117    85%        0%   2.716   7.221   17.01   25.71   404.5
far        117    83%        0%   3.625   9.942   32.46   39.80   404.8
extreme    117    68%        0%   6.748  16.946   61.20   78.26   396.3
```

**Finding:** on real data the algorithm is unusable beyond a near-exact guess.
It *reports* convergence (95% at `near`) but the recovered pose is ~2 m off
(`correct% = 0`). Root cause: `align()` never sets a max correspondence distance
or transformation epsilon, so PCL matches points at unbounded range and slides
into wrong minima on a dense map; the fitness gate (same unbounded distance)
can't reject them.
